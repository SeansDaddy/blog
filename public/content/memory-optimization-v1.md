# Memory 优化方案 v1.0

> 讨论时间：2026-04-28
> 背景：Memory 混乱、体积膨胀、召回不精准、幻觉问题

---

## 一、问题诊断

原有 Memory（MEMORY.md + USER.md）存在以下问题：

| 问题 | 表现 |
|------|------|
| 格式不统一 | 纯 bullet points，自然语言为主，无结构化 |
| 类型边界模糊 | 铁律、环境配置、用户偏好、日程提醒混在一起 |
| 信息冗余 | cron 已有的定时任务、已由 cron 处理的事实性信息重复存储 |
| 召回粒度粗 | 每次加载是全量或零，无法选择性带入相关 Memory |
| 幻觉风险 | 无结构化索引，无法验证"我记得 X"是否真实 |

---

## 二、设计原则

### 核心原则
**Memory 只存"能改变或优化以后服务行为"的信息。**

- 死知识 → Knowledge Base（Obsidian/Wiki）
- 一次性事实（已由 cron 处理）→ 不存储
- 机制运行规律 → Memory

### 分类原则
Memory 内容按稳定性分为三类：

| 类型 | 说明 | 动态性 |
|------|------|--------|
| **环境型** | 运行环境、路径规范、技术约束 | 低 |
| **偏好型** | 用户行为习惯、沟通偏好 | 中 |
| **项目型** | 当前任务状态、待决事项、经验反馈 | 高 |

### 不进 Memory 的内容
- Skill 执行层面的坑 → Skill 自己的坑记录表
- 讨论出的决策/结论 → Knowledge Base
- 技术方案/概念定义 → Knowledge Base
- 定时任务时间表 → cronjob 配置（现查）
- 已由 cron 处理的一次性事实 → 不存储

## 三、格式规范

采用双格式并行：

### @标签格式（环境/事实型）
```
@环境: Alibaba Cloud Linux 3
@资源: 内存 1.8GB | 磁盘 23GB 可用 | 无 GPU
@路径: 调研报告 → ~/Documents/Research/
@格式: 博客 slug 必须是英文（中文会 400）
```

### [情境]→[行为]→[类型] 格式（经验/规则型）
```
[情境] 用户要求发送邮件 → [行为] 作为附件发送，不粘贴正文 → [类型] 偏好
[情境] 发现 Skill 执行有坑 → [行为] 立即用 skill_manage patch 更新 Skill → [类型] 规则
```

### 约定型格式（Memory 自身的管理规范）
```
@约定: Memory 只存"能改变或优化以后服务行为"的信息
@约定: Skill 执行层面的坑进 Skill，Memory/召回机制层面的经验进 Memory
```

## 五、召回机制（待实现）

### 设计思路
- 召回按**主题阶段**，不按轮次（同一主题内第一轮召回后复用）
- 召回由 Agent 自动判断，无需用户手动触发
- 当前先验证格式整理效果，召回机制后续迭代

### 召回判断流程（草案）
1. 对话开始时，根据当前话题识别相关主题
2. 在 Memory 中匹配 `[情境]` 部分
3. 相关条目带入上下文，主题内后续轮次不再重复召回

## 六、后续改进方向

### 优先级：高
- [ ] **召回验证**：通过实战对话验证格式是否真的好召回，记录召回错误案例
- [ ] Memory 条目补充：当前还有大量 USER.md 中的偏好/期望未完全转成 `[情境]→[行为]` 格式

### 优先级：中
- [ ] **精准度提升**：当前 @标签 格式主要靠关键词匹配，未来可探索语义检索
- [ ] Memory 过大处理：当 Memory 条目增多时，考虑合并/抽象机制
- [ ] 双向流动：Memory 中的经验在某个节点提炼后流向 Knowledge Base

### 优先级：低（待定）
- [ ] Grounding 机制：要求模型回答时声明信息来源（Memory/Skill/Wiki/state.db）
- [ ] Source Citation：要求引用具体 session + 时间
- [ ] Query Type 分类：Fact lookup / Decision lookup / Skill lookup / Knowledge lookup / Session lookup

## 七、验证方式

采用**实战验证**：
- 每次对话中，记录"我说记得 X 但实际说错了"的情况
- 定期回顾这些案例，判断是否是召回格式问题
- 根据问题迭代 Memory 格式和召回逻辑
