前阵子认真研究了一下 Skill 评价这件事，发现了一个有意思的现象：业界讨论 Skill 评价时，很容易把两个完全不同的维度混在一起聊，然后得出一些看起来对但实际上没什么用的结论。

比如"Skill 好不好"，这句话其实问的是两件不同的事：
- **输出质量**：Skill 被激活之后，给出的答案好不好？
- **触发精度**：LLM 到底会不会在需要的时候激活这个 Skill？

这两个维度必须独立评估，混淆了就会出问题。

## 触发精度才是被漏掉的那个

我之前在博客里写过 promptfoo 评估输出质量，但那只是 Skill 评价的一半。另一半——触发精度——才是 Skill 能不能用、会不会被浪费的决定性因素。

想象一个场景：你的 Skill 输出质量极高，LLM 按照 Skill 里的指令执行，结果完美。但问题是，LLM 在需要调用它的时候没有调用，转而自己回答了。

这个 Skill 等于不存在。

SkillsBench 的数据印证了这一点。人工筛选的 Skill 平均能让任务通过率提升 16.2 个百分点——但这是在 Skill 被正确激活的前提下。如果没有触发，再好的 Skill 也只是文档库里的一篇 Markdown。

触发精度的核心指标是：
- **准确率**：应该触发时触发了多少次 / 总测试次数
- **误触发率**：不该触发时却触发了多少次 / 总测试次数

优化路径是一个"测试—分析—调整"的闭环：通过变更 Skill 描述（Description）的措辞，反复测试，直到触发边界清晰。

## SkillsBench 告诉我们的几件事

SkillsBench 是目前这个领域最重要的基准测试框架，86 个真实任务、11 个领域、7 种模型配置、7308 条推理轨迹。它告诉我们几个反直觉的结论：

**第一，精简的 Skill 效果最好。** 2-3 个模块的 Skill，比那种百科全书式的长文档效果高出约 4 倍。可执行代码示例是最大性能提升来源。

**第二，不是越多越好。** 每任务 2-3 个 Skill 效果最优，加更多反而收益递减甚至负向。

**第三，有 16 个任务出现了负向增益。** 这是最值得警惕的数字：Skill 不仅没有提升性能，反而让结果更差了。说明 Skill 的设计质量存在巨大差异，低质量的 Skill 不如没有。

**第四，规模可以部分被 Skill 替代。** Claude Haiku 4.5 搭配 Skills 的表现，超越了没有 Skills 的 Claude Opus 4.5。正确的程序性知识，可以弥补推理能力的不足。

## 两种东西被混在一起评价了

Skill 评价还有一个更深层的问题：很多评测框架把"Agent 的能力"和"Skill 的贡献"混在一起算。

Agent 完成一个任务，可能调用了多个 Skill，同时也依赖了自己的推理能力。单纯看任务完成率，并不能说明 Skill 本身贡献了多少。SkillsBench 的设计好就好在，它把 Skill 的贡献单独拎出来看——对比同一个模型加 Skill 和不加 Skill 的差异，这样才算公平。

这也解释了为什么自生成 Skill 没有显著收益：LLM 自己生成的 Skill，往往是在自己擅长的事情上强化，而不是在真正薄弱的环节补充。Skill 的价值在于引入外部的程序性知识，而不是放大已有的能力。

## 我们自己怎么做

基于这次研究，我觉得 Skill 评价应该分两条独立的流水线：

**流水线一：触发精度测试。** 建立四类回归样本——显式调用（直接点名 Skill）、隐式调用（描述足够强）、上下文调用（噪声场景下仍触发）、负例控制（相邻场景不误触发）。用这套样本持续测试 Description 的触发准确性。

**流水线二：输出质量评估。** 用 promptfoo 类的工具，设定确定性断言和 LLM Rubric，测试 Skill 被激活后输出的准确率、格式规范性和专业深度。

两条流水线必须都通过才能发版。触发精度不过，Skill 就是死的；输出质量不过，Skill 就是有害的。

## 一个值得记住的数字

这次研究我最印象深的数字不是 16.2pp，不是 4 倍，而是 **16**——有 16 个任务出现了负向增益。

这个数字提醒我们：Skill 不是越多越好，也不是写出来就能用。设计质量参差不齐的 Skill，不仅浪费开发资源，还可能拉低 Agent 的整体表现。评价是一件值得认认真真做的事，不是可选项。
