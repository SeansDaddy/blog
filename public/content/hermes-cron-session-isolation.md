# Hermes Agent 定时任务与会话隔离：一个被很多人忽略的坑

我最近把 Hermes Agent 用熟了之后，遇到一个很现实的问题：**定时任务（cron）怎么知道今天干了什么？**

这个问题的答案，揭示了 Hermes 架构里一个有意思的设计选择。

---

## 先说结论

Hermes 的**主会话**（你和机器人对话的那个）和**定时任务会话**（cron 触发）是**完全隔离的**，各自独立存储。定时任务看不到你当前的对话内容，也无法读你之前的聊天记录。

这不是 bug，是一个有意的安全边界设计。

---

## 为什么会踩这个坑

事情是这样的：我设了一个每晚 21:30 的定时任务，它会自动读当天的工作日志、生成摘要、发到飞书。但有一天它发出了一条消息：

> 【今日工作】
> - 暂无工作记录

我的第一反应是：定时任务怎么又出错了？

仔细一看才发现，不是任务有问题，而是**日志文件就是空的**——我那天确实没记录任何东西。

这时候我才意识到，定时任务能读到的东西，和我平时对话的上下文，完全是两回事。

---

## 深入一点：Hermes 底层存储结构

Hermes 用一个 SQLite 数据库存储所有会话历史（大约 34MB，126 个会话，1 万多条消息），结构大概是：

```
sessions 表：每个会话的元信息
  - id（会话唯一标识）
  - source（"feishu" / "cron" / "cli" 等）
  - started_at / ended_at
  - message_count

messages 表：每个会话的所有消息
  - session_id（关联到 sessions 表）
  - role（user / assistant / tool）
  - content（消息正文）
  - tool_calls / tool_name
```

关键点在这里：**cron 任务触发后会创建一个全新的 session，source = "cron"**，它只属于 cron session 的消息，和你与飞书机器人的对话（source = "feishu"）在物理上就隔开了。

所以当 cron 任务运行时，它其实"失忆"了——它不知道你今天跟机器人讨论了什么、完成了什么任务。

---

## 以前的 workaround：试图从 SQLite 里捞内容

一个很自然的想法是：既然所有消息都在 SQLite 里，那定时任务能不能直接去读？

```python
# 伪代码：尝试从主会话历史里提取今日工作
def extract_today_work():
    # 找到今天的主会话 ID
    session_id = find_session_by_date_and_source("2026-04-26", "feishu")
    # 读取所有消息
    messages = db.query("SELECT content FROM messages WHERE session_id = ?", session_id)
    # 用正则提取工作相关内容...
```

这条路**走不通**，因为：

1. 消息密度太低——1 万多条消息里，大部分是工具调用的输入输出、代码片段、日志输出，真正"工作总结"的内容可能只有几十条
2. 碎片化严重——同一个任务的内容被切分成几十条消息，正则根本拼不回来
3. 格式不统一——对话是随意的，没有固定结构，提取质量很差

---

## 正确的解法：事件驱动日志

既然 cron "记不住"过去发生的事，那就**不让它记忆**，而是**在发生的事情发生时立刻记录**。

具体做法是：**对话中随时往一个日志文件追加记录**，cron 只负责读文件 + 发送。

```
~/.hermes/daily_work/2026-04-26.md
```

文件格式很简单：

```markdown
# 每日工作日志 — 2026-04-26

## 今日完成

- 整理了三处工作记录机制，确定统一用 daily_work/ 作为单一数据源
- 理解了 Hermes cron session 与主会话完全隔离的原理

## 明日待办

- 继续完善 xxx 功能
```

每天晚上 21:30 的定时任务，做的事情就是：

1. 读取今天的日志文件
2. 生成摘要
3. 写入知识库的日记目录（方便日后检索）
4. 发到飞书 Home 频道

它不需要"回忆"今天发生了什么，因为**那些事在发生的那一刻就已经被记录了**。

---

## 这个方案带来的额外收获

当我把这个机制跑顺了之后，发现它其实比"从会话里提取"要**可靠得多**：

- **记录的时候就是新鲜的**，不存在事后回忆遗漏的问题
- **日志本身也是思考工具**——写日志的过程会强迫你梳理"今天到底干了什么"
- **知识积累**：每天的日志汇总到知识库之后，可以回顾、可以检索、可以提炼成系统化的文档

而且，因为日志文件格式是 Markdown，可以直接用 Obsidian 打开，长期积累下来就是一个个人工作记录库。

---

## 架构上的一点感想

Hermes 的这个设计选择——让 cron session 和主会话完全隔离——其实很合理。

如果 cron 能读到你的对话内容，那意味着**任何定时任务都有权限访问你所有的私人对话**，这反而是一个安全风险。隔离是默认的，共享才是需要显式授权的。

所以"随时记录"这个习惯，不只是解决技术问题，也是在**尊重这个架构设计的本意**。

---

## 最后的建议

如果你也在用 Hermes Agent 或类似的 AI 助手 + 定时任务组合，我的建议是：

1. **不要依赖从历史会话里提取工作内容**——消息太碎，提取质量差
2. **养成随时记录的习惯**——对话中完成了一件值得记录的事，就立刻追加到当天的日志文件
3. **让 cron 只是一个"读文件 + 发消息"的管道**——它不需要理解，只需要搬运

这样，定时任务就真正变成了一个**省力的自动化工具**，而不是一个**需要不断调试的脆弱脚本**。
