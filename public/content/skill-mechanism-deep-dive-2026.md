# Hermes 的 Skill 机制：从原理到执行

写这篇文章的起因是和一个朋友聊起 Agent 的 Skill 机制，他说"这不就是插件吗，装上就能用"。我说不是，他不信。后来我发现这个问题还挺普遍的，值得专门写一篇。

## 什么是 Skill

在 Hermes 的语境里，Skill 是一个**自描述的能力包**。它不是一个可执行的二进制，也不是一个被 Agent 动态加载的模块。它是一段写好的 Markdown（`SKILL.md`），配上可选的支撑文件（参考文档、模板、脚本）。

当你对着 Hermes 说"用一下那个 bilibili 下载的 Skill"，发生的事情和你想象的可能正相反——Hermes 不是去"查表找对应的代码"，而是把那个 Skill 的说明文档**读出来，塞进自己的上下文**，然后告诉你："好，我现在会了这个，我来帮你操作。"

这就是 Skill 的本质：**一次受控的上下文注入**，而不是一次代码加载。

## 触发链路：Slash Command 只是入口

触发 Skill 有两种方式：

**方式一：Slash Command。** 用户输入 `/bilibili`，Hermes 的 prompt 里有一张触发词表，匹配到了，就知道该加载哪个 Skill。

**方式二：语义推断。** 用户说"帮我下这个 B 站视频"，没说 `/bilibili`，但 Hermes 凭借对 Skill 描述的理解，自己推断出应该用 bilibili-download Skill，然后去加载它。

无论哪种方式，触发的第一步都是一样的：**根据名字找到 SKILL.md 文件，把它的内容读出来**。

## 三层加载：按需Disclosure

Skill 的内容不是一次性全部塞进上下文的。Hermes 实现了一套**渐进式披露（Progressive Disclosure）**机制，分三层：

**Tier 1 — 元数据。** 当 Hermes 刚拿到 Skill 名字时，它只知道 Skill 的简短描述、适用场景、关键约束。这是最轻的一层，用于快速判断"这个 Skill 有没有用"。

**Tier 2 — 完整 SKILL.md。** 当 Hermes 决定使用这个 Skill 时，它会把整个 SKILL.md 文件加载进来，包括步骤、命令示例、坑点记录。这一层才是执行层面的主要参考。

**Tier 3 — 支撑文件。** 如果 SKILL.md 里引用了模板、脚本、参考文档，Hermes 会按需加载它们。比如一个 Skill 引用了 `scripts/validate.py`，只有在实际执行到那一步时才会去读这个文件。

这种设计的意图是：**让上下文始终保持精简，同时在需要时能拿到完整信息**。如果你装了 50 个 Skill，Hermes 不需要把 50 个 Skill 的全部内容都塞进每次对话的上下文——它只在真正用到某个 Skill 的时候才加载它。

## 执行：没有独立的 Skill Runtime

这是最反直觉的部分。**Skill 被加载之后，执行它的是 Hermes 自己的 LLM**，而不是一个独立的执行器。

流程是这样的：

1. SKILL.md 被读出来，格式化成一段 user message（比如："你是一个 Skill 执行者，以下是你的操作步骤："）
2. 这段话和用户的原始请求一起，打包成一个新的对话消息
3. Hermes 的 LLM 接管，执行推理和工具调用
4. 执行结果返回给用户

换句话说：**Skill 不是一段被调用的代码，而是一份被转述的指令**。LLM 读完这份指令，然后用自己的理解去执行。

这就解释了为什么 Skill 的描述质量直接影响执行效果——LLM 是根据描述来理解任务的，描述写得不清楚，执行就会走偏。

## Function Calling：Skill 机制的底层能力

Skill 的触发和执行，底层依赖的是 **Function Calling**。没有这个能力，Skill 机制就是空中楼阁。

所谓 Function Calling，简单说就是：LLM 根据对话上下文，**自主决定调用哪个工具、填什么参数**。这个能力不是查表，也不是规则匹配，而是 LLM 在训练过程中学到的行为模式——给定一段对话，LLM 能够判断"用户要我做 X，我需要调用工具 Y，参数是 Z"。

Schema 在这里的作用是**告诉 LLM 有哪些工具可用、每个工具接受什么参数**。Schema 定义得清楚，LLM 就能准确地发起调用；Schema 写得模糊，LLM 就会犹豫或者调用错误。

值得注意的是，Function Calling 和 Tool Use 在实现上可能共享底层机制，但语义上有所不同。Function Calling 强调 LLM 的主动判断能力，Tool Use 则更强调工具作为 LLM 能力边界的扩展。

## 为什么这样设计

我第一次理解这个架构的时候，其实是有疑虑的——让 LLM 读指令然后执行，和直接让 LLM 处理用户请求，有什么区别？

区别在于**约束和引导**。

Skill 的 SKILL.md 是一份结构化的操作手册，包含步骤顺序、命令示例、已知的坑点。这些是 LLM 训练数据里不会有的东西，是针对具体场景的经验沉淀。LLM 读到这份文档，就相当于获得了一个**在这个场景下的专家指导**，而不是要从自己的参数知识里现场推理。

另一个价值是**可验证性**。Skill 写得好不好，可以独立评测——触发精度怎么样，执行结果对不对。这些指标可以量化，不像 prompt 优化那样玄学。

## 写在最后

Skill 不是插件，不是二进制，不是某种运行时加载的模块。它是一个被精心组织的 Markdown 文件，通过上下文注入的方式，让 LLM 在需要时"学会"一项技能，然后用自己的推理能力去执行。

理解这一点，才能理解为什么 Skill 的描述质量如此重要——它是 LLM 唯一的依据。
