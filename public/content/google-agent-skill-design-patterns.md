Google 在今年四月发了篇文章，讲 5 种 Skill 设计模式。说的是格式问题已经解决了，真正的挑战在**内容设计**——同样一份 SKILL.md，内部逻辑怎么组织，决定了 Skill 是好用还是鸡肋。

这 5 种模式不是 Google 拍脑袋想出来的，是他们观察了 30+ Agent 工具（Claude Code、Gemini CLI、Cursor 等）在 SKILL.md 格式上的实践之后，总结出来的。结论是：SKILL.md 的格式已经是共识，真正的竞争在**工作流逻辑**。

---

## 先说决策树，再逐个展开

```
你的 Skill 主要在做什么？

├── 注入领域知识（某类 API / 框架 / 规范）
│   └── Tool Wrapper
│
├── 稳定输出格式（报告 / 文档 / PR 描述）
│   └── Generator
│
├── 建立质量门禁（审查 / 评分 / 检查）
│   └── Reviewer
│
├── 模糊需求常被跳过（架构 / 规划 / 需求分析）
│   └── Inversion
│
└── 多阶段且容易跳步骤（复杂流程）
    └── Pipeline
```

---

## 模式一：Tool Wrapper

**解决什么问题**

Agent 不懂某个特定技术栈（FastAPI、Terraform、Pandas、React Server Components 等）。不需要把大量约定硬编码到系统提示词里，交给一个 Skill 来管。

**核心思路**

- SKILL.md 监听用户提示里的关键词
- 仅在命中的时候才从 `references/` 加载文档
- 把这些规则当作"绝对真理"来用

**适用场景**

- 团队内部编码规范分发
- 特定框架最佳实践注入
- 库级 API 使用约束

**关键设计点**

description 要写清楚触发关键词：比如"Use when building, reviewing, or debugging **FastAPI** applications"。只有真正开始审查或编写代码时才加载规范文档，不是默认就加载。`references/` 里放大块的领域规范，SKILL.md 只负责路由。

---

## 模式二：Generator

**解决什么问题**

Agent 每次运行生成不同文档结构，一次看着不错，十次放在一起就乱。比如同样是"写报告"，这次输出 Markdown，下次输出 HTML，第三次又是纯文本。

**核心思路**

- `assets/` 放输出模板
- `references/` 放样式指南
- SKILL.md 充当项目经理：加载模板 → 读风格指南 → 询问缺失变量 → 填充文档

**适用场景**

- API 文档生成
- 标准化提交消息
- 项目架构搭建
- 技术报告生成

**关键设计点**

SKILL.md **不包含**实际的布局或语法规则，只协调资源的检索。模板归模板，风格归风格，缺失变量单独补齐。模板里的每个章节结构必须全部出现在输出里，不能漏。

还有一个细节：Generator 必须有 **Step 1 / Step 2 / Step 3 三步加载**的概念，第一步加载背景，第二步加载格式，第三步才是生成，顺序不能乱。

---

## 模式三：Reviewer

**解决什么问题**

审查标准混在提示词里，难以替换和复用。想把 Python 风格审查换成 OWASP 安全审查？得重写整个提示词。

**核心思路**

- **检查内容**和**检查方式**分离
- SKILL.md 只规定审查协议
- 具体标准放在 `references/review-checklist.md`
- 换一套清单 = 完全不同的审查场景

**适用场景**

- 代码审查
- 安全审查（OWASP）
- Prompt 审查
- 架构方案审查

**关键设计点**

把 Python 风格清单换成 OWASP 安全清单，同一套 Skill 立刻变成安全审计工具。清单可以版本化、可替换、可按项目分层。Agent 负责应用标准，不负责临时发明标准——规则是清单里写好的，Agent 只是执行。

输出结构要固定（Summary / Findings / Score / Top 3），这样人类和程序都能处理。

---

## 模式四：Inversion

**解决什么问题**

Agent 太早开始生成。用户说"帮我设计一个系统"，它立刻画架构、选数据库、列服务，看起来完整，实际很多关键约束都没问清楚。

**核心思路**

- 颠覆"用户问 → Agent 答"的默认动态
- Agent 先当采访者，按阶段提问
- 必须拿到所有答案后才能进入最终输出
- 门控指令是"禁止继续"，不是"可以提问"

**适用场景**

- 架构设计
- 需求分析
- 迁移规划
- 安全评估
- 企业流程自动化

**关键设计点**

"DO NOT start building or designing until all phases are complete"——这句话要写在 SKILL.md 里，是明确的门控指令，不是可选项。Phase 之间有严格依赖关系：Phase 2 必须在 Phase 1 完全回答后才能开始。一次问一个问题，等待用户回答，不跳步。

Inversion 模式的 Skill 有个天然的二元错误检测：如果 Agent 还没问完问题就开始输出了，那是明确的 MAJOR 级错误，可以用 PatternValidator 来自动检测。

---

## 模式五：Pipeline

**解决什么问题**

复杂任务容易跳步骤。Agent 把前置条件跳过去，直接给一个看起来完整但实际未经验证的结果。

**核心思路**

- 用检查点强制顺序执行
- 菱形门控：用户确认上一步之前，禁止进入下一步
- 利用所有可选目录，仅在需要时才引入不同的参考文件和模板

**适用场景**

- 文档生成
- 发布流程
- 数据迁移
- 代码重构
- Incident 复盘

**关键设计点**

"Do NOT skip steps or proceed if a step fails"——这句话也要写进 SKILL.md。每一步的输入都是上一步的输出，形成完整依赖链。Gate 类型可以细分：`none`（无条件执行）、`required`（需要人工确认）、`dependency:<step_id>`（依赖前置步骤完成）。

Pipeline 模式还有一个实用的量化规则：SKILL.md 正文超过 80 行时，领域内容必须外置到 `references/`，否则维护性会急剧下降。

---

## 模式可以组合

这 5 种模式并非相互排斥，可以混合使用：

- **Pipeline + Reviewer**：Pipeline 技能在最后加一个 Reviewer 步骤，对自己的产出再做一次质量门禁
- **Inversion + Generator**：Generator 在开始时依赖 Inversion 来收集必要的变量，再填充模板
- **Pipeline + Inversion**：复杂规划任务先用 Inversion 做需求收集，再用 Pipeline 逐步执行

## 跟治理规范的关系

这 5 种模式是**内容设计的骨架**，治理规范是**内容质量的底线**。骨架对了，质量不过关还是白搭。

我们之前整理过一套 Skill 开发治理规范，核心是 8 个质量维度（安全性、功能性、合规性、易用性、可观测性、可靠性、性能效率、可维护性）和 30 条规则。设计 Skill 的时候，这两套东西要一起过：

1. 先用决策树确定用哪种模式
2. 再用治理规范检查每个模式下的实现有没有触犯底线

这两个是独立的检查维度，不是二选一。

---

## 怎么用

我的做法是这样的：

**脚手架 8 步**里，模式决策树是第二步，而且是强制步骤。先把这件事想清楚，后面写 SKILL.md 结构的时候才不会返工。

选定模式之后，按模式对应的 YAML 模板填 A-Skill内容输入.md，生成出来的 SKILL.md 结构自然是对的。最后 PatternValidator 会跑一遍模式规则，MAJOR 错误全部修完才能进入测试阶段。

整件事就变成了一条流水线：想清楚 → 填模板 → 自动校验 → 返工成本最低。
