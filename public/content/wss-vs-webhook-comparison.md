# WSS 长连接与 Webhook 回调：技术对比与选型指南

> **创建时间**：2026-04-25  
> **标签**：架构设计 · WebSocket · Webhook · 系统集成

---

在构建与第三方平台对接的系统时，消息推送方式的选择是关键架构决策之一。本文对比 WebSocket 长连接（WSS）和 Webhook 回调两种主流模式，帮你根据实际场景做出合理选择。

## 什么是 WSS（WebSocket 长连接）

WSS 是基于 WebSocket 协议的全双工通信通道。一旦建立，客户端与服务器保持持久连接，可以随时互相发送消息，无需重复建立连接和认证。

**典型特征：**
- 连接方向：客户端（OpenClaw）主动连接平台服务器，保持持久连接
- 通信方式：双向实时通信，事件推送与响应同步完成
- 连接建立后无需每次请求重新认证，无 HTTP 往返开销
- 需要定期心跳包维持连接活跃

## 什么是 Webhook

Webhook 是一种服务器主动通知机制。平台在特定事件发生时，主动通过 HTTP POST 请求将事件数据发送到预先注册的回调地址。

**典型特征：**
- 连接方向：平台服务器主动 POST 到客户端的公网端点
- 通信方式：平台触发 → 回调接收 → 通过 API 回传响应（如有需要）
- 客户端只需提供公网可访问的 HTTP 端点，无需主动连接外网
- 可靠性依赖平台的重试机制

## 核心差异对比

| 对比项 | WSS | Webhook |
|--------|-----|---------|
| **连接方向** | 客户端 → 服务器 | 服务器 → 客户端 |
| **实时性** | 双向实时推送 | 平台触发式回调 |
| **防火墙要求** | 客户端需允许出站连接 | 客户端需公网 HTTP 入口 |
| **实现复杂度** | 低（持久连接，维护简单） | 中（需公网端点 + 注册管理） |
| **认证方式** | 连接建立时一次认证 | 每次请求携带认证信息 |
| **连接维护** | 需要心跳保活 | 无连接维护开销 |
| **适用网络环境** | 客户端能访问外网 | 客户端被平台调用 |

## 架构选型建议

### 优先选择 WSS 的场景

- OpenClaw 部署在有外网访问权限的环境
- 平台提供 WebSocket 接口
- 对实时性要求较高，需要秒级甚至毫秒级响应
- 需要频繁交互的双向通信

### 选择 Webhook 的场景

- OpenClaw 部署在内网，无法主动连接外网
- 平台仅支持 Webhook 回调模式
- 客户端资源有限，不适合维护大量长连接
- 只需要接收平台推送，不需要主动查询

### 混合模式

实际项目中，两者并不互斥。同一 OpenClaw 实例可以同时支持多个平台的 WSS 连接和 Webhook 回调。例如：

- 主业务通道使用 WSS 保证实时性
- 辅助通知渠道使用 Webhook 降低复杂度

## 总结

两种模式没有绝对的优劣，只有场景的适配程度：

- **WSS**：适合"客户端主动连接、需实时交互"的场景
- **Webhook**：适合"服务端主导、客户端被动接收"的场景

选型时重点考虑网络环境、实时性需求和实现复杂度三个维度。
