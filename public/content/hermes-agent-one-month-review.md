# 我把一个 AI Agent 用进了日常工作

> 这是我用了快两个月 Hermes Agent 之后的一些真实感受。不是教程，不保证正确，纯粹个人记录。

---

## 怎么入坑的

说起来挺巧的。

那天我在找一个 Agent 平台，想给我的光伏监控系统找个自动化管理的方案。翻了几个开源项目，最后找到 Hermes Agent。吸引我的不是什么宣传语，而是一句"持久记忆 across sessions"——我对 AI Agent 最大的期待就是它能记住我上次聊到哪了，而不是每次都要我从零开始解释。

装的过程不算复杂，按文档来就行。装完之后我丢了一个问题过去："帮我查一下南京今天天气"。它回了，还告诉我是多云。

那一刻我觉得，哦，这玩意儿好像真的在听我说话。

---

## 先别想"怎么用"，先想"用什么姿势开始"

刚拿到手的时候我犯了一个错：把它当搜索引擎用。问它各种技术问题，期待它给我一个标准答案。

结果一般。

后来我换了一个思路：把它当成一个**能记住上下文的命令行工具**。不是聊天机器人，是一个长着耳朵的脚本执行器。

这个转变很关键。

当你不再期待它给你"答案"，而是让它帮你"做事情"的时候，一切都顺了。

---

## 我的几个固定用法

### 调研

我有一个固定的调研流程：遇到不熟悉的领域，先让 Agent 帮我深度搜索一下。我会告诉它我要调研什么、背景是什么、输出要包含哪些维度。它会 Spawn 子 Agent 去做并行研究，最后给我一份结构化的报告。

比如我之前要了解 Skill 治理规范的发展现状，它帮我梳理了 LangChain、OpenAI Plugin、MCP Server、CrewAI 等多个来源，输出了对比表格和参考链接。我在这个基础上再加工，比自己一个个搜快了三倍不止。

调研结果会存到 Obsidian 知识库里，下次再遇到同类问题，不用重新搜。

### 写博客

博客是我自己从零搭的，Node.js + React，部署在自家服务器上。发文章是个体力活：写正文、改 frontmatter、维护 POSTS 数组、检查标签、加到分类里。以前这些要一个个手动改，现在大部分它帮我做了。

但博客有个地方它帮不了我：中文 slug 会导致 400 错误。这个是我自己踩的坑，最后发现必须用英文 slug。类似的小坑还有排序方式（localeCompare）、归档页年份内也要排序。踩一个修一个，慢慢就好了。

### 日报

每天早上我让它跑一下日报脚本，总结昨天的工作，生成一份简短的结构化文档，存到 Obsidian 的日记目录里。不复杂，但坚持下来就成了习惯。

这大概是我现在最稳定的自动化流程之一。

## 那个 Skill 治理规范的事

有一段时间我花了比较多的精力在研究 Skill 规范上。

起因是我在做一个内部平台，要定义一套 Skill 的开发标准。平台里有很多 Agent，它们各自有各自的实现方式，我想统一一下。找了几个参考：Anthropic 的 MCP Server、LangChain 的 Tool Validator、还有 GitHub 上一个叫 speckit 的项目。

后来我把 v2.0 的规范整理了出来，存进了知识库。又补充了 v2.1 的草案，加了 PII 治理、AI Trace、Policy Engine 联动这些内容。

这个过程里我意识到一件事：**规范这种东西，写的过程中会发现很多之前没想到的地方**。写 v2.0 的时候我以为差不多了，写 v2.1 的时候发现还有六个维度没覆盖。写完之后再回头看，才知道 v1.0 有多少遗漏。

所以我现在养成了一个习惯：重要的决策先写下来，哪怕只是给自己看的草稿。写的过程就是思考的过程。

## 踩过的几个坑

说几个印象深的。

**安全扫描拦截。** 跑 cron job 的时候，脚本老是不成功。查了半天发现是 Agent 的安全组件在拦——因为配置里开了手动审批模式，终端命令全被卡住了。解决方法是把自己常用的脚本加到白名单里。这事儿花了我半小时才反应过来是这个问题。

**GitCode 图标加载不出来。** 我用了一个 CDN 托管图片，结果在博客里死活不显示。调试了半天，发现是 CDN 域名在浏览器里被拦截了。最后把图片下载到本地，问题解决。技术上的事情就是这样——你以为的解决方案，往往不是真正的问题所在。

**手机端上传。** 博客有个内容上传的功能，用手机访问的时候总是失败。我一开始以为是代码问题，后来才发现是微信内置浏览器的限制。换了系统浏览器就好了。

这些坑都不大，但加起来花了不少时间。现在回头看，这些都是"边界情况"，只有真正用了才会遇到。

## 我现在的工具链

用了一段时间之后，我的日常工作流大概是这样的：

- **信息收集**：Agent 帮我做深度调研，结果存 Obsidian
- **写作**：用 Markdown 写博客，Agent 帮我生成代码框架和结构
- **自动化**：日报、Git 备份，这些固定流程交给 cron job
- **知识管理**：Obsidian 做笔记，分了 Inbox / 学习 / 工作 / 生活几个目录
- **代码同步**：博客和知识库都有 Git 备份，push 到 GitCode

大部分事情不需要我亲自动手了。但我需要花时间想清楚：**我要做什么、做成什么样**。Agent 能帮我执行，但方向得我自己定。

## 一个不成熟的感想

用 AI Agent 的这几个月，我最大的感受是：

**它不是一个答案机器，是一个执行杠杆。**

我用它省下来的时间，主要不是"等待回答"的时间，而是"机械劳动"的时间。整理文档、搜索资料、维护代码、生成模板——这些事情它做得很稳，我不用再反复做。

但它也有边界。它不能替我决定要不要做某件事，也不能替我承担决策的后果。这些还是得自己来。

所以我现在的心态是：把执行层的事情尽量外包出去，把决策层的事情留在自己手里。这大概是人和 AI Agent 最合理的分工方式了。

---

## 接下来想做什么

有几个方向在考虑：

一个是让调研流程更自动化。现在是 Agent 帮我搜资料、整理、存到知识库，但我还要自己读一遍再加工。理想的状态是：丢一个主题过去，直接拿到一份可以直接用的报告。

另一个是想试试看能不能把光伏监控的管理端也接入 Agent。现在系统里有很多标准化的操作（查询设备状态、配置电价、生成报表），如果能用自然语言驱动，体验应该会好很多。

这些还没想清楚，先记在这里，等想明白了再动。
