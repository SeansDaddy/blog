## 背景

企业在已有成熟业务系统的情况下，引入 AI 能力通常面临两难：直接改造原系统风险高、周期长；但新建 AI 平台又要解决身份认证、数据权限、租户隔离一系列问题。

本文讨论一种**外挂式**集成思路：原生产系统零改动，AI 平台作为独立系统挂在旁边，通过已开放的接口和共享的认证体系实现无缝衔接。

## 五个关键设计点

### 1. 认证透传：不动原认证体系

原系统已有一套 Session + Cookie 认证，AI 平台不新建账号体系，而是**复用原认证**：

- Gateway 解析浏览器携带的 Session Cookie，提取用户身份
- Gateway 自签一个平台 Token 交给 OpenClaw，后续所有 Skill 调用携带此 Token
- DataProxy 收到请求后，通过 Redis 把 Token 换回原始 Cookie，再用原系统身份发起调用

对原系统来说，请求看起来来自正常浏览器，完全感知不到 AI 平台的存在。

### 2. 多租隔离：企业级 + 用户级两层

- **企业级隔离**：Gateway 根据 `tenant_id` 把请求路由到对应企业的 Agent 实例，不同企业看不到彼此的数据
- **用户级隔离**：同企业内，通过 OpenClaw 的 `dmScope: per-channel-peer` 实现用户会话隔离
- 新增用户无需管理员介入配置，Gateway 自动识别并路由

### 3. 能力边界：只读 + 建议

AI 平台定位为**纯读 + 建议**系统：

- 所有 API 调用（无论是原系统还是数据湖）均为 **SELECT 查询**，不执行任何写操作
- 诊断结论和操作建议展示给用户，但不直接生成工单、不下发控制指令
- DataProxy 层做数据过滤，防止越权数据返回

### 4. 三层并发控制

| 层次 | 位置 | 机制 |
|------|------|------|
| L1 | Redis 令牌桶 | 业务层限速，防用户级别突发 |
| L2 | Gateway rateLimit | 网关层流控，平台注入配置 |
| L3 | OpenClaw 沙箱配额 | 执行层资源隔离 |

三层各司其职，任一层失效不会导致整体失效。

### 5. 定时任务 IAM 委托

定时任务（CronTab）不能携带用户 Cookie，需要独立的权限通道：

- CronTab 通过 IAM 申请**委托 Token**，权限范围受限于定时任务本身
- 委托 Token 发给 Gateway，Gateway 验证后放行执行
- 审计日志记录所有定时任务的执行记录

## DataProxy：协议转换枢纽

DataProxy 是整个架构的数据出口，同时支持两类数据源：

- **原系统 API（HTTP）**：Token → 查 Redis → 换回 Session Cookie → 以原系统身份调用
- **数据湖（SQL）**：SQL 凭证由 DataProxy 自行管理，按需缓存，不走 Redis 存储

这种分离设计让原系统和数据湖的权限模型解耦，互不干扰。

## 小结

外挂式集成的核心思路是**复用而非替代**：复用原认证体系、复用原 API 接口、复用原权限模型。AI 平台做薄薄的一层转发和增强，原系统保持不变。

五个设计点环环相扣：认证透传解决身份问题，多租隔离解决数据安全问题，能力边界解决信任问题，三层流控解决稳定性问题，定时任务委托解决自动化场景的权限问题。
