# AI 底层逻辑概念体系：从 Token 到 Agent 的核心概念串烧

整理时间：2026-04-27

## 概念总览

| 概念 | 英文 | 说明 |
|------|------|------|
| 大语言模型 | LLM (Large Language Model) | 所有 AI 技术的核心 |
| 基本单元 | Token | 模型处理文本的最小单位，非词 |
| 上下文 | Context | 模型每次任务接收的信息总和 |
| 上下文窗口 | Context Window | Context 能容纳的最大 Token 数 |
| 提示词 | Prompt | 分 User Prompt 和 System Prompt |
| 工具 | Tools | 让模型感知和影响外部世界的函数 |
| 协议 | MCP (Model Context Protocol) | 统一工具接入标准 |
| 智能体 | Agent | 能自主规划、调用工具的程序 |
| 技能 | Agent Skill | 给 Agent 看的说明文档 |

---

## 1. LLM（大语言模型）

LLM 全称 Large Language Model，中文是大语言模型，简称大模型。

现在几乎所有大模型都是基于 **Transformer 架构**训练的：
- Transformer 由 Google 团队在 2017 年提出，论文名是《Attention is all you need》
- Google 点燃了火种，但真正引爆全世界的是 OpenAI
- 2022 年底 ChatGPT3.5 出世，是第一个达到可用级别的大模型
- 2023 年 3 月 GPT4 发布，把 AI 能力天花板拉到了新高度
- 如今不再是 OpenAI 的独角戏，Claude、Gemini 等后起之秀都在各自领域同台竞技

### 大模型的工作原理

大模型本质就是一个**文字接龙游戏**。它接收输入后，预测下一个概率最高的词/Token，输出后再抓回来追加到输入中，继续预测下一个，如此循环直到输出结束标志。

举例：
- 用户问："今天天气怎么样"
- 模型预测："非常"
- 追加到输入，再预测："适合"
- 再追加，再预测："出门"
- 直到输出结束标志
- 最终得到："今天天气非常适合出门"

关键点：理解了这一点，就明白了为什么大模型要一个词一个词地输出。

## 2. Token（词元）

### 什么是 Token

Token 是大模型处理文本的**最小基本单元**。

Token 和词并不是一对一的关系：
- "自然语言处理"按词分是 4 个，但按 Token 算可能是 3 个或更多
- "神经网络"是一个词，但可能被切成两个 Token（"神经"和"网络"）
- 英文 "Helpful" 被切成 "Help" 和 "ful" 两个 Token

Token 是模型自己学会的切分规则，切出来的每一块就是它一次能够处理的最小单位。

### 经验数据

- 一个 Token ≈ 0.75 个英文单词
- 一个 Token ≈ 1.5~2 个汉字
- 40 万 Token ≈ 60~80 万汉字 ≈ 30 万英文单词

### Tokenizer 的作用

在人类文字和大模型之间需要一个"中间人"——Tokenizer，负责：
- **编码**：把文字变成数字（Token ID）
- **解码**：把数字还原成文字

### 流程

1. 用户提问 → Tokenizer **编码**（切分 + 转 ID）→ 一串 Token ID 列表
2. 模型处理 → 输出 Token ID
3. Tokenizer **解码**（查表转文字）→ 得到回答的第一个 Token
4. 重复直到输出结束标志

## 3. Context（上下文）

### 什么是 Context

Context = 上下文，是大模型每次处理任务时接收到的**信息总和**。

包含内容：
- User Prompt（用户当前问题）
- 对话历史
- System Prompt（系统规则/人设）
- 工具列表
- 模型正在输出的每一个 Token
- 其他配置信息

### Context Window（上下文窗口）

Context Window = 上下文窗口，代表 Context 能容纳的**最大 Token 数量**。

举例：
- GPT5.4 的 Context Window：105 万 Token
- Gemini 3.1 Pro：100 万 Token
- Claude Opus 6：100 万 Token

100 万 Token ≈ 150 万汉字 ≈ 整部哈利波特全级的内容。

### RAG 技术

如果有一个超长产品手册想让大模型根据它来回答用户问题：
- 不能直接全部塞进去（太长，成本高）
- RAG（检索增强生成）可以从手册中抽取与用户问题最匹配的片段，只发给模型那几段
- 这样既不受 Context Window 限制，成本也低

## 4. Prompt（提示词）

### 什么是 Prompt

Prompt 就是用户或系统给大模型下达的**具体指令或问题**。

### Prompt 的质量很重要

Prompt 怎么写直接决定了大模型的输出质量：
- 模糊的 Prompt（如"帮我写一首诗"）→ 输出不确定
- 清晰的 Prompt（如"请帮我写一首五言绝句，主题是秋天落叶，风格要悲凉"）→ 输出更符合预期

这就是 Prompt Engineering（提示词工程）的意义——本质就是把话说清楚。

### User Prompt vs System Prompt

| 类型 | 英文 | 说明 | 配置方 |
|------|------|------|--------|
| 用户提示词 | User Prompt | 用户在对话框直接输入的问题 | 用户 |
| 系统提示词 | System Prompt | 后台配置的人设和做事规则 | 开发者 |

**举例**：数学陪练机器人
- System Prompt（开发者后台设置）："你是一个耐心的数学老师，当学生问你问题时不要直接给答案，要一步一步引导学生思考"
- User Prompt（学生输入）："三加五等于几"

有了 System Prompt，模型就知道自己扮演的是引导者角色，回答方式完全不同。

## 5. Tools（工具）

### 大模型的弱点

大模型只是文字接龙游戏，它无法感知外部世界——问"今天上海天气怎么样"，它只能回答"抱歉我无法获取实时天气数据"。

### 工具的本质

工具本质上就是一个**函数**：
- 输入：参数（如城市、日期）
- 输出：结果（如天气信息）

有了工具，大模型就能查天气、搜信息，真正感知外部世界。

### 完整调用流程

涉及四个角色：**用户 → 平台 → 大模型 → 工具**

1. 用户提问发给**平台**
2. 平台把用户问题 + 可用工具列表发给**大模型**
3. 大模型分析后生成工具调用指令（输出文本，告诉平台要调哪个工具）发回**平台**
4. **平台**真正调用**工具**
5. 工具结果返回**平台**，平台再发给**大模型**
6. 大模型归纳整理成自然语言发给**平台**
7. 平台转发给**用户**

关键点：模型唯一的能力是输出文本，它无法自己调用工具。调工具的是平台，模型只是"告诉平台它想调哪个"。

---

## 6. MCP（Model Context Protocol）

### 问题

每个平台接入工具的标准都不一样：
- 用 OpenAI 的 API → 按 OpenAI 规范接入
- 用 Claude 的 API → 按 Anthropic 规范接入
- 用 Gemini → 按 Google 规范接入

同一个工具要写三遍代码。

### MCP 的由来

能不能搞一个**统一的标准**，让所有平台都遵循？这样工具只需要开发一次，就可以在所有平台使用。

这就是 MCP（Model Context Protocol，模型上下文协议）——一套**统一的工具接入标准**。

就像所有手机都用 Type-C 接口一样，有了统一标准，大家都方便。

## 7. Agent（智能体）

### 什么是 Agent

Agent = 能**自主规划、自主调用工具、持续运作、直至完成用户任务**的程序。

### 例子：出门小助手

用户说："我马上要出门，该带些什么？"

Agent 需要自主思考并多次调用工具：
1. 调用**定位工具** → 获取用户经纬度
2. 调用**天气工具** → 获取当前位置天气
3. 判断下雨 → 调用**店铺工具** → 搜索附近便利店
4. 组合信息 → 输出最终答案

这个过程中，Agent 需要**一步一步思考当前情况，决定下一步该做什么**，而不是简单的一问一答。

### 市面产品

比较流行的 Agent 产品包括：Claude Code、Codex、Cursor 等。

构建模式五花八门：YoReAct、Planning 等。

## 8. Agent Skill（智能体技能）

### 是什么

Agent Skill 本质上是提前写好**塞给 Agent 的一份说明文档**，规定：
- 做事步骤
- 判断规则
- 输出格式
- 示例

### 为什么不直接每次说？

想象你每次出门前都要把"下雨带伞、光照强带帽子、空气差带口罩、风大穿防风外套、手机必带、按特定格式输出"全部粘贴到提问里——太反人类了。

Agent Skill 就是让你**提前写好规则，Agent 自动遵守**。

### 文档结构

分两部分：

**元数据层**（相当于封面）：
- `Name`：技能名称
- `Description`：描述这个技能是做什么的

**指令层**：
- 目标
- 执行步骤
- 判断规则
- 输出格式
- 示例（势力）

### 存放规范

以 Claude Code 为例：
- 用户目录下的 `.claude/skills/` 目录
- 每个 Agent Skill 一个子目录，**目录名必须和 Skill 名相同**
- 目录下必须有一个文件叫 **`SKILL.md`**（大写 S）
- Claude Code 启动时扫描 skills 目录，读元数据层；用户问题与 Skill 相关时才读取完整指令层

## 概念体系总结

```
LLM（大语言模型）— AI 核心引擎
  ↓
Token — 处理文本的最小基本单元
  ↓
Context（上下文）— 模型每次接收的信息总和
  ↓
Context Window — 能容纳的最大 Token 量
  ↓
Prompt（User + System）— 具体指令或问题
  ↓
Tools — 感知外部世界的函数
  ↓
MCP — 统一工具接入标准
  ↓
Agent — 自主规划 + 调用工具的程序
  ↓
Agent Skill — 给 Agent 的说明文档
```

理解了这些概念，就能看懂 AI 圈子里各种新技术新产品——无论是 Claude Code、Codex、Cursor 还是 OpenCode，本质上都是在这个框架下运作的。
